General DevOps

DevOps is a culture and practice that synergizes the efforts of development and operations teams to accelerate software delivery with a minimal failure rate. It is a value-added approach where engineers collaborate throughout the entire product lifecycle—from design to deployment—enabling continuous development, integration, testing, and monitoring.

  • Plan — Define the type of application and create a development roadmap.
  • Code — Application is coded per end-user requirements.
  • Build — Integrate various code modules from the previous step.
  • Test — Most crucial step; test and rebuild if necessary.
  • Integrate — Merge code from multiple programmers into one.
  • Deploy — Deploy to cloud; ensure changes don't disrupt high-traffic sites.
  • Operate — Perform operations on the code as required.
  • Monitor — Track performance and make changes to meet user requirements.

Technical: Continuous software delivery · Less complex problems · Early defect detection & faster correction.

Business: Faster feature delivery · Stable operating environments · Improved team collaboration.

  • Stage 1: Assess the existing process (2–3 weeks) — identify areas of improvement and create a roadmap.
  • Stage 2: Create a Proof of Concept (PoC); once approved, kick off the actual rollout plan.
  • Stage 3: Implement DevOps step by step — version control → integration → testing → deployment → delivery → monitoring.

Standard (git init): Creates a working directory + a .git subfolder containing revision history.

Bare (git init --bare): No working/checked-out copy. Revision history stored in the root folder (not .git). Used as a remote/shared repository.

git mv is used to rename or move files while keeping Git history intact.

Option 1: Fix the bad file in a new commit and push:

git commit -m "fix: revert bad change"

Option 2: Create a revert commit that undoes all changes in the bad commit:

git revert <commit-id>
# e.g. git revert 56de0938f

Git stash saves your modified tracked files onto a stack of unfinished changes so you can switch branches without committing. You can reapply them later with git stash pop.

  • You always work on the master (or main) branch by default.
  • Create a new branch to develop a feature in isolation: git checkout -b feature-name
  • Circles on a branch diagram represent individual commits.
  • After work is complete, merge the feature branch back into master.

Git Merge: Incorporates changes by creating an extra merge commit. Preserves exact history but can pollute feature branch history.

Git Rebase: Moves the base of your feature branch onto master, creating new commits for each original commit. Produces a cleaner, linear history but rewrites project history.

git diff-tree -r <commit-hash>
# e.g. git diff-tree -r 87e673f21b

The -r flag lists individual files. The commit hash lists all changed/added files in that commit.

A merge conflict occurs when two branches have competing commits and Git can't automatically decide which changes to keep.

Via GitHub conflict editor: Go to Pull Requests → click the conflicted PR → Resolve Conflicts → choose changes → Mark as resolved → Commit merge.

Via command line: Open the conflicted file, look for <<<<<<<, =======, >>>>>>> markers, edit to keep desired changes, remove markers, then:

git add .
git commit -m "Resolved merge conflict"

Continuous Integration — Jenkins

  • The Jenkins Master pulls code from a remote GitHub repository on every commit.
  • It distributes the workload to all Jenkins slaves.
  • On request from the master, slaves carry out builds, tests, and produce test reports.

A Jenkinsfile is a text file checked into source control that contains the definition of a Jenkins pipeline. It enables code review of the pipeline, provides an audit trail, and serves as a single source of truth.

java -jar Jenkins.war
  • Pipeline — User-defined CD model; defines the entire build/test/deliver process.
  • Node — A machine in the Jenkins environment capable of executing a pipeline.
  • Step — A single task telling Jenkins what to do at a specific point.
  • Stage — A conceptually distinct subset of tasks (Build, Test, Deploy).

pom.xml (Project Object Model) is used to define dependencies in Maven.

A. Scripted Pipeline — Based on Groovy DSL; uses node blocks.

node {
  stage('Build') { /* build steps */ }
  stage('Test')  { /* test steps */  }
  stage('Deploy'){ /* deploy steps */}
}

B. Declarative Pipeline — Simpler, friendlier syntax using a pipeline block.

pipeline {
  agent any
  stages {
    stage('Build')  { steps { /* ... */ } }
    stage('Test')   { steps { /* ... */ } }
    stage('Deploy') { steps { /* ... */ } }
  }
}

Periodically back up your JENKINS_HOME directory. It contains: build job configurations · slave node configurations · build history.

To clone or replicate a job, copy its directory. To rename a job, rename the directory.

  • Copy the corresponding job directory to the new installation.
  • Clone a job by duplicating the directory with a different name.
  • Rename a job by renaming its directory.
  • Internal database to store user data and credentials.
  • LDAP (Lightweight Directory Access Protocol) server.
  • Authentication mechanism used by the deployed application server.
  1. Copy the .hpi file to $JENKINS_HOME/plugins
  2. Remove the plugin's development directory
  3. Create an empty file called <plugin>.hpi.pinned
  4. Restart Jenkins

In JENKINS_HOME/config.xml, find the XML element <useSecurity> and change it from true to false. Security will be disabled on next restart.

Configuration Management — Chef, Puppet, Ansible

SSL certificates ensure each node has access to the right data between the Chef server and client. Every node has a private/public key pair — the public key is stored on the Chef server. When a certificate (containing the node's private key) is sent, the server compares it against the stored public key to identify the node and grant access.

# systemctl disable httpd.service

Test Kitchen is a command-line tool that spins up an instance and tests a cookbook on it before deploying to actual nodes.

$ kitchen create    # create instances
$ kitchen converge  # combine multiple instances
$ kitchen verify    # verify instances
$ kitchen destroy   # destroy instances
$ kitchen setup     # setup instances

chef-apply — Runs a specific recipe on the client: $ chef-apply recipe_name.rb

chef-client — Applies all cookbooks in the server's run list to the client: $ knife chef-client

# puppetca --sign hostname-of-agent
# Example: puppetca --sign ChefAgent
  • Jira — Track configuration changes; maintenance through internal procedures.
  • Git — Version control for Puppet code (with Puppet's code manager app).
  • Jenkins — Continuous integration pipeline for passing changes through.
  • Basic units of any configuration management tool.
  • Represent features of a node (software packages, services, etc.).
  • A resource declaration in a catalog describes the action to perform.
  • When a catalog executes, it sets the node to the desired state.

Classes are named blocks in a manifest that configure various node functionalities (services, files, packages). They are added to a node's catalog and executed only when explicitly invoked.

class apache (String $version = 'latest') {
  package { 'httpd': ensure => $version, before => File['/etc/httpd.conf'] }
}

An Ansible role is an independent block of tasks, variables, files, and templates embedded inside a playbook. It promotes reuse and organization of automation code.

Always use {{ }} for variables except in conditional statements (e.g., when: foo is defined) since conditionals are already run through Jinja. It helps distinguish strings from undefined variables and prevents Ansible from treating lines as dictionary declarations.

  • Roles — Manage tasks in a playbook; easily shared via Ansible Galaxy.
  • include — Add a submodule or another file to multiple playbooks.
  • import — Like include but ensures a file is added only once (prevents recursion issues).

Use Docker Swarm — a tool that creates and manages a cluster of swarm nodes. A swarm has two node types: manager node and worker nodes.

# Initialize manager node
docker swarm init --advertise-addr <MANAGER-IP>

# Add a worker node (run on the worker host with the token from init)
docker swarm join --token <TOKEN> <MANAGER-IP>:2377

Use Docker Compose. Each container runs in isolation but can interact with others. All Docker Compose files are YAML files.

A Dockerfile is used to build Docker images via the docker build command. Once built, images are uploaded to a Docker registry, and users can pull them to create containers anywhere.

Continuous Monitoring — Nagios

  • Enables server monitoring and checks if they are sufficiently utilized or have task failures.
  • Verifies the status of servers and services.
  • Inspects infrastructure health.
  • Checks if applications are working correctly and web servers are reachable.

NRPE enables executing Nagios plugins on remote Linux/Unix machines to monitor metrics like disk usage and CPU load. It consists of two parts:

  • check_nrpe plugin — Resides on the local monitoring machine.
  • NRPE daemon — Runs on the remote Linux/Unix machine.
  • Port 5666
  • Port 5667
  • Port 5668

Active Checks: Initiated by the Nagios process itself; regularly scheduled. Nagios executes a plugin, which checks the host/service state and reports back.

Passive Checks: Initiated by external applications/processes. Results are submitted to Nagios for processing via an external command file. Useful for monitoring services Nagios can't check directly.

The main configuration file contains directives that affect how Nagios operates. Both the Nagios process and CGIs read it.

/usr/local/nagios/etc/resource.cfg
  • Provides in-depth visibility into network traffic sources and security threats.
  • Gives a central view of network traffic and bandwidth data.
  • Allows admins to gather high-level health info about the network.
  • Enables proactive resolution of outages, abnormal behavior, and threats.

HTTP monitoring: Increased availability · Fast detection of outages · Web transaction & performance monitoring.

SSL certificate monitoring: Increased website availability · Frequent application availability · Increased security (alerts before certificates expire).

Nagios runs on VMware, Microsoft Virtual PC, Xen, Amazon EC2, and more. It monitors: CPU usage · Memory · Networking · VM status — ensuring quick detection of failures with reduced administrative overhead.

  • name — Template name; allows other definitions to inherit from this object.
  • use — Specifies the template object to inherit properties/variables from.
  • register — Indicates whether the object definition should be registered with Nagios [0/1].

Nagios uses an object configuration format that allows object definitions to inherit properties from other definitions, similar to OOP inheritance. Object types include: Services · Hosts · Commands · Time Periods.

  • Used for logging purposes — when enabled, Nagios watches a host/service very carefully.
  • Logs any changes it sees in the output of check results.
  • Helps in the analysis of log files for troubleshooting.
